# DemonDash Product + Build Plan

## Core Idea

Build an edgy parody app called `DemonDash` that turns fake-drill, King Von-style "crazy story" flexes, Hellcat chaos, street-persona posturing, and suspicious friend-group drama into overdramatic short-form skits inspired by King Von's cinematic storytelling.

The lane is:

- darkly funny
- cinematic storytelling with setup, tension, twist, and humble ending
- overdramatic
- embarrassing
- consequence-first
- King Von meme energy, especially Demon stare and "too intense for the situation" reactions that backfire

The lane is not:

- real gang glorification
- real drug dealing
- real violence fantasy
- "how to do crime"
- a direct clone of King Von or any real-life tragedy

King Von inspiration should come through as parody:

- vivid scene-by-scene storytelling
- "Crazy Story" pacing
- intense narrator energy
- Demon stare reaction beats
- sudden humbling twist endings

## Working Product Thesis

`DemonDash` is a parody AI skit generator where users enter a ridiculous "crazy story" scenario and get back:

- a title
- a hook
- 3 to 8 short cinematic scenes
- fake dashboard stats
- punchline ending with a humble twist
- caption cards built for reaction memes
- optional narrator script with drill-story energy
- optional export assets for TikTok/Reels/Shorts

The joke is always the same:

The user tries to channel Demon Mode confidence and drill-lore intensity, but reality hits with gas money, insurance, repo risk, court reminders, check-engine lights, fake friends, and public embarrassment.

## Product Pillars

### 1. Parody over realism

Everything should feel like a King Von "Crazy Story" parody sketch, not a real street-life simulator.

### 2. Consequences over glorification

Every flex, stare-down, loyalty test, or Demon Mode entrance must collapse into:

- no gas
- no money
- repo problems
- rental exposure
- fake friends
- legal stress
- mechanical failure
- public humiliation

### 3. Meme-speed creation

The MVP should let a user go from idea to usable skit draft in under 60 seconds.

### 4. Short-form first

Every output should be built for:

- TikTok
- Reels
- Shorts
- screenshot captions

### 5. Safety through conversion

If a user pushes into real crime, drugs, or violence, the app should rewrite the scenario into parody instead of just killing the vibe.

## Core Generation Rules

Every output should follow these writing rules:

### Rule 1

Cinematic storytelling comes first: every skit should feel like a tiny "Crazy Story" with setup, tension, character beats, twist, and humble ending.

### Rule 2

No real gang roleplay.

### Rule 3

No real drug-dealing or violence instructions.

### Rule 4

Every tough Demon Mode, stare-down, or fake-tough moment must collapse into something funny, expensive, fake, or embarrassing.

### Rule 5

The ending should humble the main character with a story twist.

### Rule 6

The voice should feel cinematic and intense, but still clearly parody.

### Rule 7

Use status cards, fake warnings, narrator-style lines, and stare-reaction moments to make it feel like a meme trailer.

## Safety Conversion System

This is one of the most important parts of the app.

## Strategy

Do not default to boring hard blocks. Convert unsafe requests into parody whenever possible.

## Pipeline

### Step 1. Classify the input

Detect whether the user is asking for:

- harmless parody
- real drug crime content
- violence fantasy
- real gang roleplay
- harassment against real people

### Step 2. Rewrite if salvageable

If the prompt can be made funny and harmless, rewrite it.

### Step 3. Reject if not salvageable

Hard-block only if the request still seeks:

- real violence praise
- instructions for crime
- realistic trafficking
- real-person threats

## LocalStack / Terraform Stack

The project now includes a real local AWS simulation stack instead of a placeholder-only plan.

Required tools:

| Required | Why |
| --- | --- |
| **Terraform** | Creates DemonDash fake AWS resources through Infrastructure as Code |
| **Docker** | Runs LocalStack locally |
| **Docker Compose** | Starts/configures LocalStack from `docker-compose.yml` |
| **LocalStack Pro image** | Simulates AWS locally with `localstack/localstack-pro:latest` |
| **LocalStack Auth Token** | Required by Docker Compose as `LOCALSTACK_AUTH_TOKEN` |
| **AWS CLI** | Verifies resources against LocalStack |
| **Terminal / PowerShell** | Runs Docker, Terraform, AWS CLI, and Next.js commands |

Created local resources:

- `demondash-dev-exports` S3 bucket for export/render preview artifacts
- `demondash-dev-projects` DynamoDB table keyed by `project_id`
- `demondash-dev-safety-events` DynamoDB table keyed by `event_id`

Runtime behavior:

- Next.js APIs use LocalStack when it is running.
- The browser UI keeps local storage fallback so the MVP still works without Docker.
- All AWS SDK and Terraform endpoints point at `http://localhost:4566`, not real AWS.

Setup reference:

- Copy `.env.example` to `.env`
- Set `LOCALSTACK_AUTH_TOKEN`
- Run `npm run localstack:up`
- Run `npm run tf:init && npm run tf:apply`
- Run `npm run aws:verify`
- Run `npm run dev`
